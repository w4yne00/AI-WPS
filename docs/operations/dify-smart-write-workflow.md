# AI-WPS 智能编写 Dify 工作流配置手册

适用版本：`v0.12.16-alpha` 及以上

适用任务：`word.smart_write`

## 1. 功能目标

智能编写用于对 WPS 任务窗口提交的选中文本执行改写润色、续写扩展、提炼总结或自定义编写。Dify 工作流负责生成最终正文，WPS 任务窗口会根据内容结构选择朴素或结构化回显：普通段落不额外套排版；标题、列表、序号、表格、加粗等结构存在时尽量正常展示和写回。

推荐调用链路：

```text
WPS 智能编写 -> 本地 adapter -> Dify Chatflow -> 大模型最终正文 -> WPS 结构感知预览
```

智能编写与审查类任务的职责不同：

| 功能 | Dify 输出 | WPS 处理方式 |
| --- | --- | --- |
| 智能编写 | 可直接替换选区的最终正文 | 结果窗口按内容结构回显；用户确认后尽量保留原选区结构写回 |
| 文档审查 | Markdown 中的 JSON 问题列表 | 结果窗口展示审查意见，不写回正文 |
| 格式审查 | 可选段落角色 JSON | adapter 按标准模板生成格式检查意见，不写回排版 |

## 2. Dify 应用配置

建议为智能编写创建独立 Chat / Chatflow 应用：

```text
开始节点 -> 大模型节点 -> 回复节点
```

adapter 对该任务使用：

```text
POST /v1/chat-messages
taskType = word.smart_write
```

任务窗口设置页配置方式：

1. `大模型 API URL` 填写 Dify API 根地址，例如 `https://aibot.chinasatnet.com.cn/v1`。
2. 在“任务级 API Key”中为“智能编写”保存该 Dify 应用的 API Key。
3. 智能编写未配置独立 Key 时，会回退使用统一 Dify Chat API Key。

当前版本的任务级 API Key 只用于选择不同 Dify 应用的认证密钥；智能编写、文档审查和格式审查仍统一调用 `/chat-messages` 路径。

## 3. 开始节点配置

开始节点建议只使用 Dify 系统默认变量：

```text
sys.query
```

无需另外创建 `source_text`、`write_action`、`style`、`focus`、`length` 或 `user_prompt` 输入字段。adapter 已将这些任务信息和原文组合为一份完整提示词，并同时传入顶层 `query` 和兼容字段 `inputs.query`。

adapter 请求体形态如下：

```json
{
  "inputs": {
    "query": "完整智能编写提示词"
  },
  "query": "完整智能编写提示词",
  "conversation_id": "",
  "response_mode": "blocking",
  "user": "wps-ai-assistant",
  "files": []
}
```

## 4. 大模型节点 SYSTEM 提示词

在大模型节点的 `SYSTEM` 提示词中填写以下内容：

```text
你是企业办公文档智能编写助手，负责根据用户提交的原文、编写动作和补充要求，生成可直接用于 Word 文档的正式中文内容。

你的任务可能包括：
1. 改写润色：保持原意和事实不变，优化措辞、逻辑、语气与可读性。
2. 续写扩展：基于已有内容继续撰写，保持上下文一致，不重复原文。
3. 提炼总结：提取关键信息、结论、风险和行动项，形成结构清晰的摘要。
4. 自定义编写：严格按照用户给出的任务要求生成内容。

工作原则：
1. 必须处理输入内容，禁止原样返回待处理原文。
2. 不得编造输入中没有提供的事实、数据、结论、责任主体、时间安排或政策依据。
3. 如输入信息不足以支持确定性结论，应采用审慎表达，例如“建议进一步确认”“需结合实际情况核实”。
4. 保持企业办公文档风格：正式、准确、简洁、条理清晰、便于直接使用。
5. 如果输入包含明确的表达风格、侧重点、篇幅或补充要求，必须优先遵守。
6. 不要说明你的处理过程，不要输出“以下是改写结果”“根据您的要求”等开场语。
7. 只输出最终可使用的正文内容。

输出格式要求：
1. 默认保持待处理原文的段落数量和换行结构，适合直接替换用户选中的段落。
2. 如果原文有多个段落，输出也应保留相近分段；不要把连续多个段落压成一整段。
3. 原文已有标题、列表、序号、表格或强调格式时，应尽量保持对应结构和层级。
4. 不要额外新增原文没有、用户也未要求的 Markdown 标题、项目符号、编号列表或表格。
5. 普通正文按自然段分段，段落之间保留原文相近的换行关系。
6. 如用户明确要求分点、编号、标题、表格或加粗，可使用对应 Markdown 结构，但不要输出代码块。
7. 不要输出 HTML 标签、JSON、XML 或调试信息。

各类任务的具体要求：

一、改写润色
- 保留原文核心事实、观点、结论和业务含义。
- 优化语句通顺度、逻辑衔接、专业表达和结构层次。
- 删除明显重复、口语化或不规范表达。
- 如用户未明确要求分点且原文不是列表，不要把普通段落改成列表；如原文已有列表，应保持列表形态。
- 不得简单复述或原样返回原文。

二、续写扩展
- 必须沿用原文语境、主题和正式程度。
- 续写内容应自然承接前文，不得重复已经表达过的内容。
- 可补充实施步骤、风险考虑、保障措施或后续计划，但不得虚构具体事实。
- 若用户明确要求分层表达，或原文已有分层结构，可使用标题或列表组织内容。

三、提炼总结
- 优先保留结论、重点事项、关键风险、责任要求和下一步安排。
- 删除铺垫、重复描述和不影响理解的细节。
- 优先采用简洁段落或编号列表。
- 如原文已有强调格式或用户要求突出重点，可使用 Markdown 加粗。

四、自定义编写
- 以用户补充要求为最高优先级，但不得违反真实性和准确性原则。
- 如果用户要求与原文冲突，应以不编造事实、不歪曲原意为前提完成表达。
- 根据输出目的和原文结构选择适合的正文结构，默认不增加额外 Markdown 排版。

最终检查要求：
- 输出内容是否确实经过处理，而非原文回显。
- 是否符合用户选择的任务类型、风格、侧重点和篇幅。
- 是否仅输出最终正文。
- 是否保持了适合替换选区的段落结构；原文已有结构是否得到保留，且未额外增加无关 Markdown 排版。
```

## 5. 大模型节点 USER 提示词

在 `USER` 消息中通过变量选择器引用系统输入变量：

```text
{{sys.query}}
```

如果现场 Dify 版本无法选择 `sys.query`，可在开始节点新增一个非必填文本变量 `query`，并在 `USER` 消息中改为引用：

```text
{{query}}
```

此兼容方案下，adapter 的 `inputs.query` 会提供同一份完整提示词。

## 6. 回复节点配置

回复节点直接绑定大模型节点正文输出，不增加自然语言前缀，不将正文转换为 JSON，不使用代码块包裹正文。

正确输出示例：

```text
当前工作中存在部分事项尚未完成闭环、关键依赖条件仍需进一步确认等问题。如依赖条件未按期落实，整体进度存在延误风险。

下一步应梳理未闭环事项，明确责任人和完成时限，形成关键依赖清单并定期跟踪落实，同时针对主要风险制定应对措施。
```

不要输出：

````text
以下是处理结果：
```text
主要问题
...
```
````

## 7. WPS 任务窗口验证

使用 `v0.12.16-alpha` 时，先完全退出并重新启动 WPS，避免旧页面缓存影响结果。

1. 打开“设置”，确认诊断区显示：

```text
前端版本：0.12.16-alpha
```

2. 进入“智能编写”，默认菜单应为“技术方案正式 / 保持信息完整 / 保持篇幅”，也可按材料场景切换为“条理化说明”“汇报材料风格”“结论与风险”“措施与计划”或“验收与闭环”。
3. 输入一段包含多个事项的原文。
4. 在补充要求中输入：

```text
请分为“主要问题”和“下一步安排”两个部分，关键风险用加粗标识，行动安排使用编号列表。
```

5. 运行后检查结果窗口：普通段落应按朴素文本显示并保留必要换行；如果原文或输出包含标题、列表、序号、表格、加粗，应正常结构化显示。

## 8. Adapter 诊断判断

执行一次智能编写后，访问：

```text
http://127.0.0.1:18100/provider/debug-last
```

应确认：

```json
{
  "taskType": "word.smart_write",
  "response": {
    "answerFormat": {
      "containsMarkdown": true,
      "containsHeading": true,
      "containsOrderedList": true,
      "containsBold": true,
      "containsParagraphBreak": true
    }
  }
}
```

判断方式：

- `containsMarkdown=false` 且 `answerLength>0`：适合普通段落场景，任务窗格会按朴素文本保留换行回显。
- `containsMarkdown=true`：适合原文或用户要求中存在标题、列表、序号、表格、加粗等结构的场景，任务窗格会自动使用结构化回显并尽量格式化写回。
- `provider=mock`：adapter 未真实调用 Dify，请检查 API URL 与智能编写任务级 API Key。

## 9. 现场诊断

设置页“最近一次任务诊断”对应 adapter 的 `/provider/debug-last`、`/provider/status`、`/provider/route-diagnostics`、`/provider/task-api-keys`。诊断信息只显示脱敏摘要，不显示完整原文和 API Key。

如果前台结果异常，优先确认：

1. `taskType` 是否为 `word.smart_write`。
2. `authSource` 或 `taskAuthSource` 是否为任务级密钥文件。
3. `url` 是否为统一 API URL 拼接 `/chat-messages`。
4. `request.bodyKeys` 是否包含 `inputs`、`query`、`response_mode`、`user`。
5. `response.answerLength` 是否大于 0。

## 10. 注意事项

- 智能编写当前以“选区段落替换”为主。普通段落优先保持朴素正文形态；标题、列表、序号、表格、加粗等结构存在时，预览会结构化展示，写回会做 best-effort 格式化。
- WPS 宿主不支持对应格式化 API 时，写回会自动降级为结构化文本，例如保留编号、项目符号或 Markdown 表格文本，避免任务失败。
