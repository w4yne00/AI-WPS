# AI-WPS 智能编写 Dify 工作流配置手册

适用版本：`v0.12.9-alpha` 及以上

适用任务：`word.smart_write`

## 1. 功能目标

智能编写用于对 WPS 任务窗口提交的原文执行改写润色、续写扩展、提炼总结或自定义编写。Dify 工作流负责生成最终正文，WPS 任务窗口负责将 Dify 返回的 Markdown 渲染为可阅读的预览效果。

推荐调用链路：

```text
WPS 智能编写 -> 本地 adapter -> Dify Chatflow -> 大模型 Markdown 正文 -> WPS Markdown 预览
```

智能编写与审查类任务的职责不同：

| 功能 | Dify 输出 | WPS 处理方式 |
| --- | --- | --- |
| 智能编写 | 可直接使用的 Markdown 正文 | 结果窗口渲染；用户确认后写回正文 |
| 文档审查 | Markdown 中的 JSON 问题列表 | 结果窗口展示审查意见，不写回正文 |
| 格式审查 | 可选段落角色 JSON | adapter 按标准模板生成格式检查意见，不写回排版 |

## 2. Dify 应用配置

建议为智能编写创建独立 Chat / Chatflow 应用：

```text
开始节点 -> 大模型节点 -> 回复节点
```

adapter 对该任务使用：

```text
POST /v1/chat-messages
taskType = word.smart_write
```

任务窗口设置页配置方式：

1. `大模型 API URL` 填写 Dify API 根地址，例如 `https://aibot.chinasatnet.com.cn/v1`。
2. 在“任务级 API Key”中为“智能编写”保存该 Dify 应用的 API Key。
3. 智能编写未配置独立 Key 时，会回退使用统一 Dify Chat API Key。

当前版本的任务级 API Key 只用于选择不同 Dify 应用的认证密钥；智能编写、文档审查和格式审查仍统一调用 `/chat-messages` 路径。

## 3. 开始节点配置

开始节点建议只使用 Dify 系统默认变量：

```text
sys.query
```

无需另外创建 `source_text`、`write_action`、`style`、`focus`、`length` 或 `user_prompt` 输入字段。adapter 已将这些任务信息和原文组合为一份完整提示词，并同时传入顶层 `query` 和兼容字段 `inputs.query`。

adapter 请求体形态如下：

```json
{
  "inputs": {
    "query": "完整智能编写提示词"
  },
  "query": "完整智能编写提示词",
  "conversation_id": "",
  "response_mode": "blocking",
  "user": "wps-ai-assistant",
  "files": []
}
```

## 4. 大模型节点 SYSTEM 提示词

在大模型节点的 `SYSTEM` 提示词中填写以下内容：

```text
你是企业办公文档智能编写助手，负责根据用户提交的原文、编写动作和补充要求，生成可直接用于 Word 文档的正式中文内容。

你的任务可能包括：
1. 改写润色：保持原意和事实不变，优化措辞、逻辑、语气与可读性。
2. 续写扩展：基于已有内容继续撰写，保持上下文一致，不重复原文。
3. 提炼总结：提取关键信息、结论、风险和行动项，形成结构清晰的摘要。
4. 自定义编写：严格按照用户给出的任务要求生成内容。

工作原则：
1. 必须处理输入内容，禁止原样返回待处理原文。
2. 不得编造输入中没有提供的事实、数据、结论、责任主体、时间安排或政策依据。
3. 如输入信息不足以支持确定性结论，应采用审慎表达，例如“建议进一步确认”“需结合实际情况核实”。
4. 保持企业办公文档风格：正式、准确、简洁、条理清晰、便于直接使用。
5. 如果输入包含明确的表达风格、侧重点、篇幅或补充要求，必须优先遵守。
6. 不要说明你的处理过程，不要输出“以下是改写结果”“根据您的要求”等开场语。
7. 只输出最终可使用的正文内容。

输出格式要求：
1. 输出必须使用 Markdown 格式，以便 WPS 任务窗口正确展示排版效果。
2. 普通正文按自然段分段，段落之间保留空行。
3. 需要标题时，使用 Markdown 标题，例如：
   # 一级标题
   ## 二级标题
   ### 三级标题
4. 需要列举事项、步骤、要求或措施时，使用 Markdown 有序列表或无序列表。
5. 需要强调重点结论、风险、关键要求或行动项时，使用 Markdown 加粗，例如：**重点内容**。
6. 仅在内容确实适合比较展示时使用 Markdown 表格。
7. 不要输出 HTML 标签。
8. 不要把最终正文包裹在 Markdown 代码块中。
9. 不要输出 JSON、XML 或调试信息。

各类任务的具体要求：

一、改写润色
- 保留原文核心事实、观点、结论和业务含义。
- 优化语句通顺度、逻辑衔接、专业表达和结构层次。
- 删除明显重复、口语化或不规范表达。
- 如原文适合分段或分点，可转换为 Markdown 段落或列表。
- 不得简单复述或原样返回原文。

二、续写扩展
- 必须沿用原文语境、主题和正式程度。
- 续写内容应自然承接前文，不得重复已经表达过的内容。
- 可补充实施步骤、风险考虑、保障措施或后续计划，但不得虚构具体事实。
- 若适合分层表达，可使用标题或列表组织内容。

三、提炼总结
- 优先保留结论、重点事项、关键风险、责任要求和下一步安排。
- 删除铺垫、重复描述和不影响理解的细节。
- 优先采用简洁段落或编号列表。
- 如存在明确重点，可使用加粗标识。

四、自定义编写
- 以用户补充要求为最高优先级，但不得违反真实性和准确性原则。
- 如果用户要求与原文冲突，应以不编造事实、不歪曲原意为前提完成表达。
- 根据输出目的选择适合的 Markdown 结构。

最终检查要求：
- 输出内容是否确实经过处理，而非原文回显。
- 是否符合用户选择的任务类型、风格、侧重点和篇幅。
- 是否仅输出最终正文。
- 是否使用了适当的 Markdown 分段、编号和加粗格式。
```

## 5. 大模型节点 USER 提示词

在 `USER` 消息中通过变量选择器引用系统输入变量：

```text
{{sys.query}}
```

如果现场 Dify 版本无法选择 `sys.query`，可在开始节点新增一个非必填文本变量 `query`，并在 `USER` 消息中改为引用：

```text
{{query}}
```

此兼容方案下，adapter 的 `inputs.query` 会提供同一份完整提示词。

## 6. 回复节点配置

回复节点直接绑定大模型节点正文输出，不增加自然语言前缀，不将正文转换为 JSON，不使用代码块包裹 Markdown。

正确输出示例：

```markdown
## 主要问题

当前工作中存在以下需要重点关注的问题：

1. 部分事项尚未完成闭环，可能影响后续实施安排。
2. 关键依赖条件仍需进一步确认。
3. **如依赖条件未按期落实，整体进度存在延误风险。**

## 下一步安排

1. 梳理未闭环事项，明确责任人和完成时限。
2. 形成关键依赖清单，并定期跟踪落实状态。
3. 针对主要风险制定应对措施。
```

不要输出：

````text
以下是处理结果：
```markdown
## 主要问题
...
```
````

## 7. WPS 任务窗口验证

使用 `v0.12.9-alpha` 时，先完全退出并重新启动 WPS，避免旧页面缓存影响结果。

1. 打开“设置”，确认诊断区显示：

```text
前端版本：0.12.9-alpha
```

2. 进入“智能编写”，默认菜单应为“技术方案正式 / 保持信息完整 / 保持篇幅”，也可按材料场景切换为“条理化说明”“汇报材料风格”“结论与风险”“措施与计划”或“验收与闭环”。
3. 输入一段包含多个事项的原文。
4. 在补充要求中输入：

```text
请分为“主要问题”和“下一步安排”两个部分，关键风险用加粗标识，行动安排使用编号列表。
```

5. 运行后检查结果窗口是否显示标题层级、分段、编号列表和加粗文本。

## 8. Adapter 诊断判断

执行一次智能编写后，访问：

```text
http://127.0.0.1:18100/provider/debug-last
```

应确认：

```json
{
  "taskType": "word.smart_write",
  "response": {
    "answerFormat": {
      "containsMarkdown": true,
      "containsHeading": true,
      "containsOrderedList": true,
      "containsBold": true,
      "containsParagraphBreak": true
    }
  }
}
```

判断方式：

- `containsMarkdown=true` 且前端版本正确，但任务窗格仍无样式：重点检查 WPS 是否仍持有旧任务窗口，关闭任务窗口并重新从 Ribbon 打开。
- `containsMarkdown=false`：Dify 实际输出不包含 Markdown，请检查 SYSTEM 提示词是否已保存、回复节点是否直接绑定大模型正文。
- `provider=mock`：adapter 未真实调用 Dify，请检查 API URL 与智能编写任务级 API Key。

## 9. 注意事项

- Markdown 格式只作用于任务窗口预览；点击应用预览写入 Word 时，当前版本写入的是模型正文文本，不会将 Markdown 标记自动转换为 Word 标题或加粗样式。
- 如果下一步需要将 Markdown 内容写回 Word 后也保留真实标题、列表和粗体，需要新增 Markdown 到 WPS 文档对象格式的写回能力，属于独立功能改造。
