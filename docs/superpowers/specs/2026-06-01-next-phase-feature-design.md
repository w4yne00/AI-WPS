# AI-WPS 下一阶段功能设计规划

日期：2026-06-01

基线版本：`v0.12.11-alpha`

适用范围：基于已验证通过的稳定增强版，规划下一阶段功能方向。本文只定义设计边界、优先级和验收标准，不包含实现计划。

## 1. 当前基线

当前版本已经将 Word 侧任务稳定收敛为三条业务链路：

- 智能编写：面向选中文本或全文内容生成改写、续写、总结等结果。
- 文档审查：面向技术方案、合同验收文档、测试大纲及细则，检查错别字、语言逻辑、通畅性和专业性问题。
- 格式审查：面向标准模板做格式问题检查，不执行自动排版写回。

当前 adapter 采用统一 Dify `/chat-messages` 调用方式：

- 请求正文同时携带顶层 `query` 和 `inputs.query`。
- 任务级 API Key 只决定认证密钥，不改变接口 path、payloadStyle 或输出解析路径。
- 三个任务分别使用独立任务级 API Key，可命中各自 Dify 工作流。
- 结果展示侧支持 Markdown 安全渲染和最近一次任务诊断。

## 2. 下一阶段设计原则

下一阶段不应破坏已经验证通过的稳定链路，设计上遵循以下原则：

- 不改动智能编写、文档审查、格式审查现有 adapter 路由和任务级 API Key 选路逻辑。
- 不恢复旧版多路径 workflow 路由，不引入新的 Dify payload 风格。
- 避免一次版本同时推进多个大功能，优先选择一个主线形成可验证闭环。
- 继续适配 Dify 只能输出 Markdown 的限制，避免强依赖 JSON 输出模式。
- 长文档处理必须按选区、分段或局部任务设计，不能假设 Dify 和模型能稳定处理整篇长文档。
- 任何写回 Word 的能力都必须先从低风险、可预测格式开始，不做大模型自由生成的整篇自动排版写回。

## 3. 候选功能方向

### 方向 A：智能编写写回增强

目标：让智能编写返回的 Markdown 结果能够以更接近 Word 文档的格式写回正文，而不仅是纯文本插入。

核心能力：

- 将 Markdown 标题写回为 Word 标题段落。
- 将有序列表、无序列表写回为 Word 列表或近似列表文本。
- 将段落分隔保留为 Word 段落。
- 将加粗等基础样式映射到 Word 字符格式。
- 保留当前“复制”和“结果预览”的稳定行为。

价值：

- 用户在智能编写后可以直接得到更可用的 Word 内容。
- 对现有智能编写链路增益最大，体验提升明显。

风险：

- WPS JS/COM 对 Word 样式写入能力需要逐项验证。
- Markdown 到 Word 的样式映射需要控制边界，不能支持过多复杂语法。

推荐程度：高。

### 方向 B：文档审查闭环

目标：让文档审查从“看结果”升级为“处理问题”，形成审查、定位、采纳、记录的闭环。

核心能力：

- 按问题类型分组展示审查结果。
- 每条问题支持复制建议、复制改写、标记已处理。
- 支持根据问题位置提示用户回到对应段落。
- 生成本次审查记录摘要，便于归档或粘贴到审查表。
- 保留选区审查和全文审查，继续规避长文档上下文限制。

价值：

- 更贴合企业文档审查流程。
- 能把大模型输出转化为可执行的审查清单。

风险：

- 自动定位段落的准确性受 WPS 文档结构和模型输出位置描述影响。
- 第一阶段应避免自动修改正文，先做人工确认闭环。

推荐程度：高。

### 方向 C：格式审查半自动修复

目标：在当前格式审查基础上，对确定性格式问题提供一键应用能力。

核心能力：

- 只对明确规则项提供修复，例如纸张、页边距、正文字体、字号、行距、段前段后。
- 对标题层级、表题、图题等依赖语义识别的项目，先保留审查建议，不自动写回。
- 所有修复前展示预览摘要，用户确认后执行。
- 修复后重新运行格式审查，生成前后差异摘要。

价值：

- 比“智能排版”更可控，降低大模型上下文和输出长度限制带来的失败率。
- 能逐步接近“按模板排版”的目标。

风险：

- 需要更深入操作 WPS Word 对象模型。
- 必须设计撤销和失败保护，否则容易影响用户文档。

推荐程度：中。建议在 A 或 B 稳定后推进。

### 方向 D：部署运维增强

目标：提升内网目标机部署、联调和问题排查效率。

核心能力：

- 设置页导出当前脱敏诊断报告。
- 一键检测 adapter、provider、任务级 API Key、Dify 调用路径。
- 生成目标机自检包，包括脚本输出、版本号、配置摘要、最近一次任务诊断。
- 增强交付包中的运维脚本说明和常见错误处理。

价值：

- 降低现场配置和排错成本。
- 对多台内网终端部署更友好。

风险：

- 用户可感知功能提升不如 A/B 明显。
- 需要持续维护诊断字段和脚本兼容性。

推荐程度：中。适合作为每个版本的附带增强，不建议单独作为大版本主线。

## 4. 推荐路线

推荐下一阶段采用“一个主线 + 一个轻量辅助增强”的方式推进：

### 首选主线：方向 A，智能编写写回增强

原因：

- 当前智能编写已经验证稳定，Dify 链路风险最低。
- Markdown 渲染已经在任务窗口侧打通，下一步自然延伸到 Word 写回。
- 用户每天使用频率高，体验收益直接。
- 功能边界可以控制在基础 Markdown，不必一次支持完整 Markdown 规范。

建议目标：

- 第一版只支持段落、标题、列表、加粗四类格式。
- 不处理表格、图片、代码块、复杂嵌套列表。
- 保留纯文本写回兜底。
- 所有写回动作必须由用户点击确认触发，不自动覆盖正文。

### 备选主线：方向 B，文档审查闭环

原因：

- 如果当前团队更重视审查流程和质量管理，B 的业务价值更明显。
- B 不需要复杂 Word 样式写入，技术风险低于 C。
- 可以沉淀企业文档审查的可追踪结果。

建议目标：

- 第一版只做问题处理状态和审查记录生成。
- 不做自动修改正文。
- 不依赖模型返回 JSON 模式，继续从 Markdown 中解析结构化片段或以 Markdown 分组展示。

### 暂缓主线：方向 C，格式审查半自动修复

原因：

- 它最接近“自动排版”，但也最容易影响正文和版式。
- 需要更完整的 WPS Word 对象模型验证。
- 应先通过 A 中的 Markdown 写回能力积累写入经验，再进入格式规则写回。

## 5. 建议版本拆分

### v0.13.0：智能编写写回增强

范围：

- Markdown 基础语法解析。
- Word 写回基础格式映射。
- 写回失败时自动降级纯文本。
- 增加写回前预览说明和结果提示。

验收：

- 智能编写原有预览、复制、Dify 调用均不回退。
- 标题、段落、列表、加粗在 Word 中有可见格式差异。
- 非支持语法不会导致写回失败或任务窗口卡死。

### v0.13.1：文档审查处理闭环

范围：

- 问题卡片处理状态。
- 复制建议和复制改写。
- 生成审查记录。
- 结果区交互和可读性优化。

验收：

- 选区审查和全文审查继续可用。
- 审查结果可以形成可复制的处理记录。
- 不改变文档正文，避免误改风险。

### v0.13.2：格式审查确定性修复试点

范围：

- 仅支持页面设置、正文段落字体字号、行距等确定性规则。
- 用户确认后才应用。
- 修复失败时显示明确错误和诊断信息。

验收：

- 小文档和中等文档能稳定应用规则。
- 不依赖 Dify 输出长 JSON。
- 执行后可重新审查并看到问题减少。

## 6. 重点保护边界

后续实现过程中必须保护以下既有逻辑：

- `word.smart_write`、`word.document_review`、`word.format_review` 三个任务类型和任务级 API Key 映射。
- provider 调用统一使用 `/chat-messages`，不恢复旧的 `/workflows/run` 或多 payloadStyle 自动路由。
- `/provider/debug-last` 继续脱敏，不记录完整密钥和完整文档内容。
- 任务窗口 Markdown 渲染继续做 HTML 转义和危险链接拦截。
- WPS 选区、全文读取、段落抽取的限量保护不能取消。
- 格式审查继续保持只读审查，不在未确认设计前恢复自动排版写回。

## 7. 需要优先确认的问题

进入实现计划前，需要先确认下一阶段主线：

- 选择 A：优先提升智能编写写回 Word 的格式保真度。
- 选择 B：优先完善文档审查的问题处理闭环。
- 选择 C：优先试点格式审查的确定性一键修复。
- 选择 D：优先增强目标机部署和诊断能力。

建议选择 A 作为下一版主线，B 作为后续版本主线，D 作为每个版本的辅助增强，C 在 A/B 稳定后再推进。

## 8. 成功标准

下一阶段功能完成后，应满足：

- 已验证稳定的智能编写、文档审查、格式审查基本功能不回退。
- Dify 工作流配置方式不变，用户不需要重新理解一套新的 adapter 路由规则。
- 新功能可以在目标机通过明确步骤验收。
- 出错时用户能在设置页诊断中看到任务类型、provider 状态、任务级 API Key 命中情况和失败摘要。
- 交付包、README、handoff 和对应 Dify 操作手册同步更新。
