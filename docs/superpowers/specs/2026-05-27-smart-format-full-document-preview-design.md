# Smart Format Full Document Preview Design

更新时间：2026-05-27

目标版本：`v0.12.2-alpha`

## 问题

智能排版接收完整 Word 文档，但 adapter 生成 Dify 段落角色识别提示词时仅包含前 120 个非空段落。长文档中第 121 段及之后的段落不会进入大模型识别，只能依赖本地规则回退，复杂结构容易识别不足。

同时，任务窗口目前仅展示需要修改格式的条目，不显示实际扫描覆盖范围。用户提交整篇文档后，看到的局部变更列表容易被理解为系统只处理了部分正文。

## 已确认方案

采用“全文分批角色识别 + 变更计划预览 + 覆盖统计提示”：

1. WPS 继续一次提交整篇文档的段落与结构信息到 `/word/format-preview`。
2. adapter 对所有非空段落按最多 120 段一批生成角色识别请求，依次调用智能排版 Dify 应用并合并合法角色结果。
3. 单批调用失败、返回无法解析或遗漏某个段落时，该段落继续使用既有本地规则兜底；整篇预览不因某批失败而中断。
4. 本地模板规则仍是最终格式来源，大模型只提供段落角色，不改写正文，不生成 WPS 命令。
5. 任务窗口仍只列出需要应用的格式变更项，但在顶部明确显示全文扫描段落数、AI 识别段落数、本地兜底段落数，并说明列表不是正文摘录。

## 数据流

```text
WPS 全文段落
  -> POST /word/format-preview
  -> 按 120 个非空段落切分批次
  -> word.smart_format /chat-messages (每批 JSON 角色识别)
  -> 合并已识别角色 + 缺失段落本地规则兜底
  -> 模板 targetProperties 变更计划
  -> 任务窗口显示覆盖统计及待应用格式项
  -> 用户确认后写回 Word 格式
```

## 接口调整

`/word/format-preview` 原有 `changes` 字段保持不变，`summary` 增加：

| 字段 | 含义 |
| --- | --- |
| `paragraphCount` | 本次扫描的非空段落总数 |
| `aiClassifiedParagraphCount` | Dify 返回并被接受的段落角色数量 |
| `localFallbackParagraphCount` | 采用本地规则判定角色的段落数量 |
| `aiBatchCount` | 实际尝试发送给 Dify 的批次数量 |

原有 `changeCount` 表示需要写回的格式变更项数量，可能小于 `paragraphCount`，因为已经符合模板的段落无需写回。

## 前端展示

排版预览顶部展示：

```text
模板：技术文件格式及书写要求
全文扫描段落：245
待调整项：131
AI 识别段落：245 | 本地兜底段落：0
识别来源：enterprise-dify-chat/task-file

以下仅显示需要调整的格式项，正文内容不会在预览中改写。
```

若连接旧版 adapter 未提供新增摘要字段，任务窗口仍兼容显示原有模板、变更数和来源，不报错。

## 保护边界

- 保持智能排版“预览后应用”，不静默覆盖文档。
- 不改变正文文本内容，只应用模板格式属性。
- 不修改智能编写、格式校对、技术文档审查的数据流和 UI。
- 不更改任务级 API Key 选择规则，也不恢复按任务路由 path 的逻辑。

## 测试

- formatter 单测构造超过 120 段的文档，验证第二批段落能获得 AI 角色并进入目标样式变更计划。
- formatter 单测验证 `summary` 的全文覆盖统计。
- 前端布局冒烟验证任务窗口包含全文扫描与本地兜底提示文本。
- 执行现有智能编写、格式校对、技术文档审查及打包回归测试。
