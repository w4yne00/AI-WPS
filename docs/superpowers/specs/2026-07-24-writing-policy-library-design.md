# Word 写作规范库设计

日期：2026-07-24

状态：设计已确认，等待按实施计划开发

来源决策：[`CONTEXT.md`](../../../CONTEXT.md) 与 [`docs/adr/`](../../adr/)

取代文档：[`2026-07-16-enterprise-terminology-style-knowledge-design.md`](./2026-07-16-enterprise-terminology-style-knowledge-design.md)

## 1. 背景

Word 插件的智能编写、智能仿写和文档审查需要稳定遵循组织专有名词、技术材料文体和去模板化表达规则。现有企业术语与风格库原型只覆盖用户自建规则，缺少可随正式安装交付的基础规范、可追溯来源、分层覆盖、规范升级和结果检查机制。

本版本被视为写作规范库的第一次正式安装基线，不承担旧版 `enterprise_knowledge` 数据库或接口的生产迁移责任。设计必须从第一次安装开始保证：后续升级只更新预置基线，不覆盖用户已经创建的术语、文体规则、去模板化规则、组织覆盖和停用记录。

参考项目固定为 `w4yne00/yangqi-tech-writing` 的 Git 标签 `v1.1.0`，提交 `d3640165569071251248a5fafb2def6ef2fe2cf4`。只提炼适用于短文本模型任务的写作核心，不复制其完整工作流，也不使用 `Humanizer-zh`。

## 2. 目标

- 统一以“写作规范库”管理术语规范、文体规则和去模板化规则。
- 随正式安装包交付一组小而可审查的预置规范包。
- 允许组织对预置项进行覆盖、停用和恢复，并创建自定义规范。
- 为智能编写、智能仿写和文档审查提供确定、受限、单次模型调用的规范注入。
- 在结果中提供非阻断的本地规范检查，突出需要人工核对的改动。
- 支持 CSV/XLSX 导出、修改、预览和回导。
- 采用紧凑、下钻式、Apple Design 取向的 Word 设置和任务界面。
- 确保未来安装升级保留组织数据，并在规范库异常时保持三项 Word 任务可用。

## 3. 非目标

- 不改智能编写、智能仿写、文档审查的写回、复制、预览、超时、轮询和工作流选择链路。
- 不改格式审查。
- 不为 Excel 或 PPT 增加写作规范库入口或运行时注入。
- 不新增第二次模型调用，不新增 Dify 工作流变量。
- 不判断作者身份，不声称检测“AI 生成”，不执行技术正确性、法律适用性或事实真实性判断。
- 不实现完整证据台账、多文档追踪、项目上下文推断、个人风格模仿或运行时联网更新。
- 第一版不支持第三方规范包安装。

## 4. 领域模型

### 4.1 规范类型

| 类型 | 作用 | 典型示例 |
|---|---|---|
| 术语规范 | 固定标准写法、别名、禁用写法与适用上下文 | 产品名、系统名、网络安全术语 |
| 文体规则 | 约束语气、句式、结构和责任表达 | 技术文件、党政公文写法 |
| 去模板化规则 | 减少空泛、堆叠、机械转折和模板式表达 | 删除无信息量铺垫、避免同义反复 |

“去模板化”只描述表达质量，不使用“去 AI 味”作为产品术语，也不对文本来源作判断。

### 4.2 分层模型

生效规范由以下四层确定：

1. **预置基线**：随版本发布、只读、带版本和来源。
2. **组织覆盖**：以稳定预置项 ID 为键，替换预置项的可编辑内容。
3. **组织自定义规范**：用户独立创建的术语或规则。
4. **预置停用记录**：组织明确停用某一预置项。

组织层优先于预置层。编辑预置项不会修改安装目录内的 JSON，而是创建组织覆盖；“恢复预置”删除对应覆盖或停用记录。

### 4.3 适用范围

文体规则和去模板化规则使用两个独立的多选维度：

- 任务范围：智能编写、智能仿写、文档审查。
- 场景范围：G企技术材料、网络安全技术材料、党政公文。

新建规则默认适用于全部任务和全部场景。术语规范默认全局生效，再通过上下文关键词减少无关匹配。

### 4.4 固定优先级

从高到低：

1. 原文受保护内容，包括专有名词、数字、日期、责任主体和明确条款。
2. 用户本次明确指令或仿写模板意图。
3. 组织覆盖与组织自定义规范。
4. 预置文体与术语基线。
5. 通用去模板化规则。

同一层级出现冲突时，不让模型自行猜测。解析器按确定规则选择一个结果，并在诊断中记录冲突；涉及受保护内容时在结果中显示“需要核对”。

## 5. 预置规范包

第一版只显示并交付当前存在的四个包：

| 规范包 | 内容 | 默认用途 |
|---|---|---|
| G企技术写作基础 | 组织化、正式、可执行的通用技术材料规则 | 自动匹配的基础层 |
| 网络安全术语 | 网络安全常用标准术语、别名和禁用写法 | 网络安全场景 |
| 技术文件文体 | 可研、方案、规范、汇报等技术文件表达规则 | G企与网络安全场景 |
| 党政公文文体 | 正式公文常用结构和表达约束 | 党政公文场景 |

首版目标规模为 80 至 120 条，经去重和人工审阅后确定：

- G企技术写作基础：15 至 20 条。
- 技术文件文体：15 至 25 条。
- 党政公文文体：10 至 15 条。
- 网络安全术语：40 至 60 条。

数量不是验收目标。每一项必须有稳定 ID、清晰适用范围、可追溯来源和明确的人工审阅结论。

## 6. 来源与审阅门禁

### 6.1 来源约束

- `yangqi-tech-writing` 只从干净标签读取：`git show v1.1.0:<path>`。
- 不从本地脏工作树 `/Users/wayne/Documents/guoqi-write-style` 提取内容。
- 固定记录标签、提交哈希和 MIT 许可证。
- 政府公文和网络安全规范记录来源名称、版本或标准号。
- 规则内容采用面向产品的原创归纳，不批量复制标准原文。

### 6.2 提取范围

纳入短文本写作核心：

- G企技术材料的正式、克制、可执行表达。
- 原文事实、数字、责任、条款和专有名词保护。
- T1/T2/T3 去模板化规则。
- 与三项 Word 任务直接相关的场景规则。
- 术语规范及可确定的别名、禁用写法。

排除：

- 证据台账、项目上下文和多文档溯源。
- 技术正确性、法律适用性和外部事实核验。
- 完整文档阶段推断。
- 个人写作风格模仿。
- 作者身份或 AI 生成判断。

### 6.3 人工审阅清单

候选项先生成 CSV 和 XLSX 审阅清单，至少包含：

- 稳定 ID
- 规范包
- 类型与分类
- 名称
- 规则或术语内容
- 正例与反例
- 来源、版本和定位
- 默认启用状态
- 审阅决定
- 审阅备注

生成器执行重复 ID、近似重复、互斥规则、空字段和无来源检查。只有明确标记为“通过”的候选项才能进入启用的预置 JSON。草稿审阅文件不进入正式交付包。

## 7. 预置文件格式

每个规范包是一个版本化 JSON 文件：

```json
{
  "schemaVersion": 1,
  "packId": "yangqi-tech-writing-base",
  "version": "1.0.0",
  "name": "G企技术写作基础",
  "sceneIds": ["yangqi", "cybersecurity"],
  "source": {
    "name": "yangqi-tech-writing",
    "version": "v1.1.0",
    "commit": "d3640165569071251248a5fafb2def6ef2fe2cf4",
    "license": "MIT"
  },
  "entries": []
}
```

术语项字段：

```json
{
  "id": "term.cyber.zero-trust",
  "type": "term",
  "category": "网络安全",
  "preferredText": "零信任",
  "aliases": ["零信任架构"],
  "forbiddenVariants": [],
  "definition": "以持续验证为核心的安全理念",
  "contextKeywords": ["身份", "访问控制", "认证"],
  "priority": 60,
  "defaultEnabled": true,
  "sourceRefs": ["source-id-or-standard"]
}
```

规则项字段：

```json
{
  "id": "rule.style.responsibility-explicit",
  "type": "style",
  "category": "责任表达",
  "name": "明确责任主体",
  "ruleText": "涉及实施、检查或整改时，应明确责任主体。",
  "positiveExample": "运维部门负责完成整改并复核。",
  "negativeExample": "相关部门做好整改。",
  "taskTypes": ["smart_write", "smart_imitate", "document_review"],
  "sceneIds": ["yangqi", "cybersecurity", "official"],
  "contextKeywords": ["负责", "整改", "落实"],
  "priority": 70,
  "defaultEnabled": true,
  "sourceRefs": ["yangqi-tech-writing:v1.1.0:reference"]
}
```

`anti_template` 与 `style` 使用相同结构，仅 `type` 不同。加载器严格拒绝未知类型、重复 ID、无效范围和超出长度限制的字段。

## 8. 组织数据与升级契约

### 8.1 数据位置

- 数据库：`run/writing_policies.db`
- 备份：`run/writing_policies.db.backup-<timestamp>`
- 预置包：安装目录内只读 JSON
- 数据库和备份权限：仅当前服务用户可读写

### 8.2 数据表

| 表 | 责任 |
|---|---|
| `schema_metadata` | 数据库 schema 版本和迁移状态 |
| `organization_terms` | 组织自定义术语 |
| `organization_rules` | 组织自定义文体和去模板化规则 |
| `preset_overrides` | 预置项覆盖或停用记录 |
| `policy_imports` | 导入预览、确认和审计元数据 |

`preset_overrides` 使用稳定 `preset_entry_id` 关联预置项，操作类型为 `override` 或 `disabled`。覆盖内容可以保存为经过类型校验的 JSON，以支持术语和规则联合类型；未经当前 schema 校验的内容不得入库。

### 8.3 升级规则

1. 安装程序不得删除、替换或初始化覆盖已有 `writing_policies.db`。
2. 新版本只替换预置 JSON、来源清单和应用代码。
3. 数据库迁移只能向前、可重复执行，并在迁移前创建备份。
4. 导入操作在应用前创建备份。
5. 最多保留最近三份自动备份。
6. 预置项 ID 一经发布不得改变语义；删除项需在包清单中保留废弃映射。
7. 新版预置与组织覆盖合并时，组织数据始终优先。
8. 数据库损坏或迁移失败时不得自动重建空库。

## 9. 生效规范解析

### 9.1 场景选择

Word 三项任务显示一个紧凑选择器，并按任务分别记忆：

- 自动匹配
- G企技术材料
- 网络安全技术材料
- 党政公文
- 不使用写作规范

自动匹配采用保守策略：

- 明确命中网络安全上下文时使用基础、技术文件和网络安全包。
- 明确命中党政公文上下文时使用基础和党政公文包。
- 明确命中一般 G企技术材料时使用基础和技术文件包。
- 无法可靠判断时只使用 G企技术写作基础。

选择“不使用写作规范”只禁用本次规范注入和结果检查，不修改数据库或任务工作流配置。

### 9.2 合并与排序

解析顺序：

1. 加载所选场景的预置快照。
2. 应用预置停用记录。
3. 应用组织覆盖。
4. 加入适用的组织自定义规范。
5. 按任务范围和场景范围过滤。
6. 对术语执行精确别名、禁用写法和上下文匹配。
7. 对规则执行任务、场景和上下文匹配。
8. 按“组织层、上下文命中、优先级、稳定 ID”确定性排序。

同样输入、同样规范快照必须产生同样的生效规范和排序。

### 9.3 注入预算

- 沿用现有 3000 字符提示词补充上限。
- 最多注入 30 个术语匹配。
- 最多注入 8 条非术语规则，优先保留 5 条文体规则和 3 条去模板化规则；某类不足时另一类可使用空余槽位。
- 受保护内容约束始终排在注入块首位。
- 不向模型注入完整规范包。
- 达到预算时按确定性排序截断完整条目，不截断半条规则。

## 10. 三项 Word 任务集成

### 10.1 单次模型调用

前端请求新增可选字段 `writingPolicyScene`，默认 `auto`。adapter 在现有模型请求之前本地解析生效规范，将规范块并入原有 `query`，仍只调用一次当前工作流。

不得新增 Dify 输入变量。provider 内部参数统一命名为 `writing_policy_block`，取代旧设计的 `enterprise_knowledge_block`。

### 10.2 智能编写

- 保护原文明确事实、数字、日期、责任主体、条款和专有名词。
- 注入命中的术语、文体和去模板化规则。
- 用户本次指令高于通用写作规范，但不得覆盖受保护内容。
- 结果继续使用现有预览、纯文本、对照、复制和写回能力。

### 10.3 智能仿写

- 仿写模板的句式与结构意图高于预置文体。
- 受保护内容和组织术语仍保持最高约束。
- 去模板化规则不得破坏用户明确要求保留的模板结构。
- 结果继续使用现有预览、纯文本和复制能力，不新增对照或写回。

### 10.4 文档审查

- 保持现有长任务、轮询、恢复和超时策略。
- 将术语偏差、文体偏差和模板化表达作为附加审查问题。
- 只输出问题、原因和建议，不修改 Word 原文。
- 不将规范检查错误解释为模型连接失败。

## 11. 本地结果规范检查

模型返回后执行轻量、确定性的本地检查：

- 受保护项发生变化：显示高风险“需要核对”。
- 术语禁用写法或标准写法偏差：显示“需要核对”。
- 明确命中的 T1/T2/T3 表达模式：显示低风险“表达建议”。
- 文档审查将检查项合并到原有问题列表。
- 无问题时只显示一行“已完成写作规范检查”。

检查不阻断结果预览、复制或写回。不得因为本地检查失败丢弃模型结果。运行日志只记录规则 ID、计数、耗时和状态，不记录规则正文、API Key 或用户全文。

任务响应可选返回：

```json
{
  "writingPolicyUsage": {
    "enabled": true,
    "scene": "cybersecurity",
    "presetVersion": "1.0.0",
    "termCount": 8,
    "styleRuleCount": 3,
    "antiTemplateRuleCount": 2,
    "truncated": false,
    "degraded": false
  },
  "writingPolicyAudit": {
    "needsReview": [],
    "expressionSuggestions": []
  }
}
```

本版本不保留 `/enterprise-knowledge/*` 别名，也不返回旧 `knowledgeUsage` 字段。

## 13. CSV/XLSX 往返

### 13.1 导出

支持：

- 导出当前生效规范。
- 仅导出组织自定义、覆盖和停用记录。

导出必须保留预置项 ID、来源、版本、层级、覆盖状态、任务范围和场景范围。XLSX 和 CSV 使用同一列定义，CSV 为 UTF-8。

### 13.2 回导语义

- 修改预置行：创建或更新组织覆盖。
- 将预置行标记停用：创建预置停用记录。
- 将预置行标记恢复：删除组织覆盖或停用记录。
- 新增无预置 ID 的行：创建组织自定义规范。
- 删除导出文件中的行不表示删除数据库记录，必须使用明确操作列。

### 13.3 安全门禁

1. 解析并严格校验文件。
2. 显示新增、修改、停用、恢复、冲突和错误预览。
3. 用户明确确认。
4. 创建数据库备份。
5. 在单个事务中应用。

预览令牌短期有效并绑定文件摘要。应用失败时完整回滚。

## 14. Word 交互设计

### 14.1 任务界面

- 在现有工作流选择器附近显示单个紧凑的写作规范选择器。
- 三项任务分别记忆选择，不跨任务覆盖。
- 默认“自动匹配”，不增加首屏说明卡。
- 结果区只显示一行规范摘要；点击后展开命中项和检查建议。
- “需要核对”与“表达建议”使用文字、图标和颜色共同区分。
- 不改变现有生成按钮、结果预览、复制和写回位置。

### 14.2 设置入口

Word 设置首页增加一个“写作规范库”入口。进入后使用分段控件：

- 预置规范
- 组织规范

预置规范按包下钻到条目；编辑预置项创建组织覆盖，并提供“恢复预置”。组织规范按术语、文体、去模板化类型筛选和编辑。导入、导出、备份和诊断放入“更多”或二级页面。

### 14.3 Apple Design 约束

- 沿用 Word 蓝色，不改变 Word、Excel、PPT 各自主色。
- 使用系统字体、清晰层级、渐进披露和紧凑间距。
- 圆角不超过 8px，不使用卡片套卡片。
- 长选项卡可水平滚动，后续新增功能时不压缩文字。
- 操作提供即时状态反馈；危险操作要求明确确认。
- 动画轻量并尊重 reduced-motion。
- 窄任务窗中不得出现横向页面溢出、文字遮挡或固定高度截断。

## 15. 性能、降级与隐私

### 15.1 性能目标

- 本地规范解析和结果检查在开发机目标不超过 100ms。
- 麒麟 V10 目标机验收目标不超过 200ms。
- 前端不得同步遍历完整规范库或渲染全部条目。
- 条目列表分页或虚拟化，搜索采用防抖。
- 规范功能不增加模型请求次数，不改变现有模型超时。

以上是工程验收目标，不是当前已测量事实。

### 15.2 规范降级

预置包、数据库、迁移或解析异常时：

1. 跳过不可用的规范增强。
2. 继续执行原有 Word 任务。
3. 前端显示“写作规范暂未应用，已继续处理”。
4. 诊断记录脱敏错误类型、阶段和时间。
5. 不自动创建空库覆盖原数据。

### 15.3 隐私

- 规范库全程保存在本地 adapter。
- 只把本次命中的有限规范与原任务内容一起发给当前模型后台。
- 日志不得记录 API Key、完整用户文本、完整规则正文或导入文件内容。
- 导出文件由用户主动生成，不自动上传。

## 16. 安装与交付

正式交付包包含：

- 四个已审阅的预置 JSON。
- 预置规范清单、版本、来源和许可证说明。
- CSV/XLSX 模板。
- 写作规范库使用说明和升级保护说明。
- 更新后的 adapter、Word 插件和测试记录。

安装脚本第一次安装时允许创建 `run/writing_policies.db`。后续覆盖安装必须检测现有数据库并原样保留。预置包随应用版本替换，组织数据库不进入交付归档，也不得由升级脚本覆盖。

## 17. 受保护的既有逻辑

实施时必须保持：

- 智能编写现有格式保真、段落恢复、对照高亮和写回行为。
- 智能仿写现有预览、纯文本和复制行为。
- 文档审查现有长任务、轮询、恢复、审查记录和只读行为。
- 三项任务现有 think 内容过滤。
- 现有工作流档案、API URL、API Key 和任务级工作流选择。
- 格式审查全部行为。
- Excel 智能分析和 PPT 智能总结全部行为。
- 三宿主 Ribbon 隔离和统一安装包。
- 现有模型超时、adapter 自启动和安装升级保护。

## 18. 验收标准

- 四个预置包通过 schema、重复、冲突、来源和人工审阅门禁。
- 三项 Word 任务可选择场景并只执行一次模型调用。
- “自动匹配”不确定时只启用基础包。
- 组织覆盖、自定义、停用和恢复在重启后保持。
- 覆盖安装后组织数据逐字节保留或经受控迁移保持等价。
- CSV/XLSX 导出、修改、预览和回导语义正确，失败完整回滚。
- 受保护项变化显示“需要核对”，T1/T2/T3 显示“表达建议”。
- 规范库异常时三项 Word 任务继续工作，且显示中文降级提示。
- 格式审查、Excel、PPT 和写回链路的回归测试无变化。
- Word 任务和设置界面在目标窄窗中无溢出、遮挡和不可达操作。

## 19. 置信度与待验证项

- **分层覆盖、稳定 ID 和独立组织数据库可满足升级保护：高。**
- **单次模型调用加本地检查能在不改变链路的前提下提升一致性：高。**
- **80 至 120 条首版规模足以覆盖高频场景：中。** 需要候选项去重和真实材料试用后确认。
- **自动场景识别准确度：中。** 第一版以保守降级到基础包控制误用风险。
- **麒麟 V10 本地解析 200ms 目标：中。** 需在目标机用真实规则数量验收。
