# Word 企业术语与写作规范知识库设计

日期：2026-07-16

状态：**已被取代，仅保留为历史记录**

替代文档：[`2026-07-24-writing-policy-library-design.md`](./2026-07-24-writing-policy-library-design.md)

说明：后续开发统一使用“写作规范库”领域名称、`writing_policy` 模块、`/writing-policies/*` API 与 `run/writing_policies.db` 数据库。本文件中的 `enterprise_knowledge`、`knowledgeUsage`、旧库迁移和兼容接口设计不再作为实施依据。

## 1. 背景与目标

Word 插件的智能编写、智能仿写和文档审查需要稳定识别公司专有名词、标准写法和企业写作风格。目前这些约束主要依赖固定提示词或用户临时输入，无法持续维护，也无法保证不同 Dify 工作流版本使用同一套企业规范。

本功能新增 adapter 本地企业知识层，用于：

- 统一维护公司名称、产品名称、系统名称、缩写、别名和禁用写法。
- 维护企业写作风格规则，并为规则提供推荐写法和不推荐写法。
- 在智能编写和智能仿写时，把本次相关的企业知识注入现有提示词。
- 在文档审查时检查术语和风格偏差，只输出问题和建议，不自动写回 Word。
- 控制企业知识注入长度，避免中型规则库加剧 Dify 最大 Token 和慢模型问题。
- 保持 API Key 工作流切换与企业知识相互独立，切换新旧 Dify 工作流时企业规范不变。

## 2. 已确认决策

- 采用“全局企业库 + 功能补充库”。
- 预计规模为 100 至 1000 条术语和若干写作风格规则。
- 支持标准 Excel 模板和 UTF-8 CSV 批量导入，同时支持前台逐条维护。
- 智能编写、智能仿写生成时遵循企业知识；文档审查检查企业知识偏差。
- 功能补充库按任务模块绑定，不按 API Key 工作流档案绑定。
- 风格规则采用“规则 + 推荐写法 + 不推荐写法”。
- 公司专有名词以全局标准为最高优先级；功能补充风格可以覆盖全局风格。
- 任务结果显示简洁知识命中反馈，明细默认收起。
- 采用 adapter 本地结构化匹配，不采用全量提示词注入或 Dify 知识库检索。
- 知识库异常时继续执行模型任务，但必须明确显示“企业知识未应用”，不得静默降级。
- 新增界面采用下钻式设计，每一层只处理一个主要任务。

## 3. 设计原则

### 3.1 确定性优先

企业专有名词属于确定性数据。标准写法、别名和禁用写法由 adapter 精确匹配，不交给模型推断，也不依赖向量检索相似度。

### 3.2 Token 可控

不把整个知识库塞入提示词。adapter 只注入本次文本命中的术语、始终应用规则和与当前任务相关的高分风格规则。

### 3.3 与工作流解耦

企业知识不绑定具名 API Key 工作流档案。用户切换 Dify 工作流版本时，只切换模型认证档案，不切换企业知识。

### 3.4 保持现有业务链路

本版本不修改：

- Word 选区和文档范围读取逻辑。
- `word.smart_write`、`word.smart_imitation`、`word.document_review` 任务类型。
- 新旧 Dify `/chat-messages` 输入兼容。
- API URL、工作流档案和 API Key 路由。
- 文档审查后台任务、长轮询和恢复机制。
- 智能编写结果预览、对照、复制和现有回写实现。
- 智能仿写的预览、纯文本和复制能力，不增加对照或写回。
- 文档审查问题状态、复制建议和审查记录能力。

## 4. 总体架构

新增 `EnterpriseKnowledgeService`，位于 adapter 层。一次任务的数据流为：

1. Word 前端按现有方式提交选中文本和任务参数。
2. adapter 任务入口识别当前 `taskType`。
3. 企业知识服务读取全局库和当前功能补充库。
4. 匹配器根据本次文本、用户要求和任务类型筛选术语与风格规则。
5. 现有 Prompt Builder 在正文前加入独立的“企业术语与写作规范”区块。
6. adapter 按现有方式向 Dify 提交完整 `query`。
7. 现有结果解析继续处理模型输出，并在结果中附加 `knowledgeUsage` 元数据。
8. Word 结果区显示简洁命中摘要，原预览和回写行为保持不变。

企业知识区块不新增 Dify 用户输入变量，避免要求现场逐个修改工作流开始节点。

## 5. 本地存储

### 5.1 存储位置

固定使用：

```text
adapter-start-kit/run/enterprise_knowledge.db
```

使用 Python 标准库 `sqlite3`，不新增运行依赖。

### 5.2 数据表

#### `knowledge_terms`

| 字段 | 说明 |
| --- | --- |
| `id` | 稳定 UUID |
| `scope` | 首版固定为 `global` |
| `category` | 公司、产品、系统、模块、接口、缩写等 |
| `preferred_text` | 标准写法 |
| `aliases` | 别名列表 |
| `forbidden_variants` | 禁用或不推荐写法列表 |
| `definition` | 简要说明 |
| `context_keywords` | 上下文关键词列表 |
| `priority` | 高、中、低 |
| `enabled` | 是否启用 |
| `note` | 维护备注 |
| `created_at` / `updated_at` | 审计时间 |

首版只允许在 `global` 维护术语硬标准。功能补充库不得覆盖同一术语的全局标准写法。

`preferred_text` 规范化后必须全局唯一；别名和禁用写法不得与其它术语的标准写法、别名或禁用写法冲突。

#### `style_rules`

| 字段 | 说明 |
| --- | --- |
| `id` | 稳定 UUID |
| `scope` | `global`、`word.smart_write`、`word.smart_imitation` 或 `word.document_review` |
| `name` | 规则名称 |
| `rule_text` | 规则正文 |
| `positive_example` | 推荐写法 |
| `negative_example` | 不推荐写法 |
| `context_keywords` | 适用关键词列表 |
| `always_apply` | 是否始终应用 |
| `priority` | 高、中、低 |
| `enabled` | 是否启用 |
| `note` | 维护备注 |
| `created_at` / `updated_at` | 审计时间 |

功能补充风格规则与全局风格冲突时，以当前功能规则为准。

同一 `scope` 下的风格规则名称规范化后必须唯一。

#### `knowledge_imports`

记录导入时间、文件名、格式、总行数、新增数、更新数、冲突数、错误数和导入结果。不保存上传文件正文。

## 6. 批量导入与导出

### 6.1 支持格式

- 标准模板 `.xlsx`。
- UTF-8 `.csv`。

XLSX 由 adapter 使用 `zipfile` 和 XML 标准库严格解析模板首个工作表，不支持公式、宏、合并单元格和任意自由格式工作簿。这样无需引入 `openpyxl`，并减少目标机离线依赖变化。

首版导入安全预算固定为：文件最大 5 MB、最多 5000 条数据行、单个单元格最多 2000 个字符、XLSX 解压后总量最多 20 MB。超限时拒绝整个文件并返回中文校验错误。

### 6.2 标准模板列

```text
类型
适用范围
名称
标准写法/规则
别名/禁用写法
推荐示例
不推荐示例
关键词
优先级
始终应用
启用
备注
```

列表字段使用固定分隔符，导入说明必须明确转义规则。

### 6.3 导入流程

1. 用户进入独立“批量导入”页面并选择文件。
2. adapter 完成文件类型、表头、字段、范围和重复项校验。
3. 前端显示新增、更新、冲突和错误统计。
4. 全局术语标准冲突不得自动覆盖，必须由用户保留库内标准或取消该行。
5. 用户可先导入全部无冲突项。
6. 写入使用单个 SQLite 事务，失败时整体回滚。
7. 每次成功导入前自动备份数据库，最多保留最近 3 份。

### 6.4 导出

首版只提供按当前范围导出 UTF-8 CSV，并提供数据库完整备份下载；不提供 XLSX 导出。

## 7. 匹配与注入

### 7.1 匹配输入

- 智能编写：待处理原文、用户补充要求。
- 智能仿写：模板、仿写需求、参考素材。
- 文档审查：本次待审查文本、文档类型和审查重点。

### 7.2 术语匹配

- 对标准写法、别名和禁用写法做规范化精确匹配。
- 同一条术语只注入一次。
- 记录实际命中片段，但诊断接口不返回完整原文上下文。
- 文档审查中，别名或禁用写法作为 `professional` 类问题交给模型输出标准建议，不修改现有问题 JSON 顶层结构。

### 7.3 风格匹配

风格评分依次考虑：

1. 当前任务范围。
2. `always_apply`。
3. 关键词命中。
4. 规则优先级。
5. 功能补充规则优先于全局风格规则。

### 7.4 默认注入预算

- 术语最多 30 条。
- 风格规则最多 8 条。
- 企业知识区块最多 3000 个中文字符。
- 术语优先于风格。
- 精确命中优先于仅关键词相关。
- 高优先级优先于中、低优先级。
- 超限时按完整规则截断，不注入半条规则。
- 截断数量写入诊断和 `knowledgeUsage`。

这些默认值应集中定义，便于后续根据目标机 Dify 上下文限制调整，但首版前台不暴露复杂的预算设置。

### 7.5 Prompt 区块

新增独立区块示例：

```text
企业术语与写作规范（必须遵守）：

[术语]
- “卫星网管平台”应统一写为“卫星互联网运营管理平台”。

[风格]
- 结论先行，避免宣传化和绝对化表述。
  推荐：已完成总体方案设计，具备下一阶段实施条件。
  避免：全面领先，完美实现所有建设目标。
```

智能编写的段落和结构保持约束仍然拥有同等约束力。企业知识不得要求模型新增原文不存在的标题、列表或表格。

## 8. Adapter 接口

新增以下本地接口：

```text
GET    /enterprise-knowledge/summary
GET    /enterprise-knowledge/items?scope=...&type=...&query=...
POST   /enterprise-knowledge/items
PATCH  /enterprise-knowledge/items/{itemId}
DELETE /enterprise-knowledge/items/{itemId}
GET    /enterprise-knowledge/import-template.csv
GET    /enterprise-knowledge/import-template.xlsx
POST   /enterprise-knowledge/imports/preview
POST   /enterprise-knowledge/imports/apply
GET    /enterprise-knowledge/export.csv?scope=...
GET    /enterprise-knowledge/backup
GET    /enterprise-knowledge/diagnostics
```

前端使用 JSON 请求上传导入预览文件，字段固定为 `fileName`、`mimeType`、`sizeBytes` 和 `contentBase64`。adapter 先校验声明大小与 Base64 解码后的实际大小，再进入 CSV/XLSX 解析；导入预览接口的 HTTP 请求体上限固定为 7 MB，以容纳 5 MB 原文件的 Base64 开销，其它接口继续沿用现有限制。

导入预览返回 10 分钟有效的一次性预览令牌，正式写入时使用该令牌，避免前端重复传送和重新解析大文件。原始上传文件只在请求内存和临时解析阶段存在，预览完成后立即删除。

模板下载接口由 adapter 使用同一份列定义生成 UTF-8 CSV 和无宏标准 XLSX，确保页面下载的模板与解析器始终一致。`export.csv` 只导出业务条目；`backup` 下载 SQLite 完整数据库快照，两者不得包含 API Key。

现有三个 Word 任务接口不改变请求字段。响应结果增加可选字段：

```json
{
  "knowledgeUsage": {
    "applied": true,
    "degraded": false,
    "termMatchCount": 6,
    "styleRuleCount": 3,
    "truncatedCount": 0,
    "matchedItems": [
      {"id": "...", "type": "term", "name": "卫星互联网运营管理平台"}
    ]
  }
}
```

`matchedItems` 最多返回 20 条，只包含条目 ID、类型和名称，不返回完整规则、示例或原文命中上下文。完整命中数量仍通过计数字段返回。

旧 adapter 或旧结果没有 `knowledgeUsage` 时，前端按现有行为显示，不报错。

## 9. Word 交互设计

### 9.1 下钻层级

#### 第一层：设置首页

新增“企业术语与写作规范”摘要，只显示条目总数、启用数、最近更新时间和“进入管理”。

#### 第二层：知识库范围

只显示四行：

- 全局企业规范。
- 智能编写补充。
- 智能仿写补充。
- 文档审查补充。

批量导入作为页面底部独立入口。

#### 第三层：范围详情

- 只保留“术语 / 风格”两段切换。
- 提供搜索、一个新增图标和一个更多菜单。
- 列表行点击进入详情，不在每行堆叠修改、停用、删除按钮。
- 更多菜单只放下载模板、导出和批量操作。

#### 第四层：单项编辑

- 使用全宽独立页面。
- 首屏只显示标准写法或规则、别名/示例和说明。
- 分类、关键词、优先级、始终应用和启用状态放入“高级设置”。
- 删除放在页面底部并二次确认。

批量导入也使用独立页面，按“选择文件、校验、处理冲突、应用”下钻。

### 9.2 任务结果反馈

- 智能编写和智能仿写显示“企业知识：已应用 X 条术语、Y 条风格规则”。
- 文档审查显示“企业知识：已检查 X 条术语、Y 条风格规则”。
- 命中明细默认收起，用户展开后显示规则名称和标准写法。
- 不增加生成前确认步骤。
- 不改变现有结果视图、复制、审查记录或智能编写回写按钮。

## 10. 异常与降级

知识库文件不存在、损坏、读取失败或匹配异常时：

1. 现有模型任务继续执行。
2. 不注入企业知识区块。
3. 结果区明确显示“企业知识未应用，本次结果仅使用模型工作流生成”。
4. `knowledgeUsage.applied=false`、`degraded=true`，并返回用户可读的短原因。
5. 最近一次任务诊断记录脱敏错误码、阶段和恢复建议。
6. 设置页知识库摘要显示异常，并提供“检查知识库”入口。

不得静默降级，也不得把数据库路径、堆栈或完整选中文本直接显示给普通用户。

导入预览失败不修改数据库；导入应用失败必须回滚事务并保留原数据库。

## 11. 安全与配置保护

- 企业知识数据库不保存 API Key。
- 管理接口只监听现有本地 adapter 地址并复用当前本地访问边界。
- 导入文件执行 5 MB 文件、5000 行、2000 字符单元格和 20 MB 解压总量预算。
- XLSX 拒绝宏、外部关系、路径穿越和异常压缩比。
- HTML 展示继续使用安全转义，知识条目不得作为原始 HTML 渲染。
- 诊断只记录命中数、截断数、耗时、条目 ID 和脱敏错误。
- 覆盖安装必须备份并恢复：

```text
run/enterprise_knowledge.db
run/enterprise_knowledge.db.backup-*
```

- 现有 `config/adapter.json`、统一 API Key 和 `run/provider_api_keys/` 的保护逻辑保持不变。

### 12.1 自动化测试

- SQLite 建库、升级、CRUD、唯一约束和事务回滚。
- 全局术语冲突和功能风格覆盖规则。
- CSV 与标准 XLSX 解析、无效文件和安全预算。
- 术语别名、禁用写法、关键词和优先级匹配。

- 三类 Prompt Builder 的企业知识注入位置和原约束保留。
- 三个任务结果的 `knowledgeUsage` 兼容。

- 420x900 和 320x700 任务窗尺寸。

- 低频高级字段默认折叠。
- 导入冲突和错误状态清晰但不拥挤。

### 12.3 麒麟 V10/WPS 真机验收

- 使用真实公司术语测试标准写法、别名和禁用写法。
- 智能编写保持原文段落、列表、标题和表格结构。
- 智能仿写同时遵守模板结构和企业风格。
- 文档审查把术语和风格偏差列入专业性问题，不自动写回。
- Dify 慢模型、think 模式和长任务恢复保持正常。
- 切换新旧具名工作流后企业知识不变化。
- 覆盖安装后 API URL、全部 API Key 和企业知识数据库均保留。
- 数据库故障时任务继续，前台明确显示未应用。

## 13. 非目标

首版不包含：

- Dify 知识库、向量数据库或语义 RAG。
- 从任意 Word/Markdown 制度文件自动提取规则。
- 项目级独立知识方案下拉。
- 用户角色、多人审批和远程知识库同步。
- 自动修改文档审查发现的问题。
- 修改智能编写既有回写策略。
- 把企业知识绑定到 API Key 工作流档案。

## 14. 发布与兼容

- 版本号和正式交付包在实施计划中确定。
- 单一 Word/Excel/PPT 正式包继续覆盖安装三个宿主。
- 企业知识管理入口首版只出现在 Word 设置页；Excel 和 PPT 不显示该入口。
